# CLAUDE.md — 삼국지 영웅 키우기

## 1. 프로젝트 정체성

**목표: 한국에서 가장 중독성 높은 삼국지 영웅 키우기 모바일 PWA를 만든다.**

- **협업 방식**: 수석(사용자)이 단계별 프롬프트로 지시하고, 선임(Claude)이 구현한다.
  큰 방향 결정은 수석의 몫이며, 선임은 구현안·리스크·대안을 보고하고 승인받아 진행한다.
- **언어**: 모든 소통과 게임 내 텍스트는 한국어. 코드 식별자는 영어.
- **확정된 핵심 결정** (2026-07-15, 2026-07-17 갱신):
  - 플랫폼: 모바일 웹 PWA (2026-07-17 수석 확정 — 기존 "godot 4" 표기는 오기였음)
  - 코어 루프: 방치형 + 가챠
  - 저장: 로컬 우선, 서버 이전 가능한 세이브 구조
  - 아트: UI는 CSS/SVG, 영웅 초상은 AI 생성 PNG(치비 스타일, 개발 시 `tools/generate-heroes.mjs`로 생성해 정적 에셋 포함 — 런타임 외부 요청 없음)
  - 스택: 바닐라 JS ES모듈, 빌드 도구 없음

## 2. 게임 디자인 헌장

### 코어 루프
> 방치 전투로 재화 획득 → 영웅 뽑기·성장 → 더 강한 스테이지 돌파 → 더 많은 재화

모든 신규 기능은 이 루프를 강화하는지 먼저 검증한다. 루프와 무관한 기능은 수석 승인 없이 추가하지 않는다.

### 중독성 5대 장치 (모든 화면·기능 설계의 기준)

1. **즉각 보상** — 모든 탭·클리어·획득에 0.1초 내 반응(숫자 카운트업, 파티클, 진동감 있는 연출). 유저 입력에 침묵하는 UI는 버그로 취급한다.
2. **복귀 보상** — 오프라인 동안 쌓인 재화를 복귀 시 한 번에 크게 지급. "끄고 있어도 이득"이라는 감각이 재접속을 만든다. 오프라인 계산은 타임스탬프 기반으로 정확하게.
3. **가챠 연출** — 등급별 차등 연출(빛 색상·긴장감 단계). 천장(피티) 시스템으로 좌절을 방지하고, 천장까지 남은 횟수를 항상 보여줘 "조금만 더"를 유도한다.
4. **수집 욕구** — 삼국지 영웅 도감. 세력별(위·촉·오·군웅) 컬렉션 보너스로 "모으면 강해진다"를 보장. 미보유 영웅은 실루엣으로 노출해 갈증을 만든다.
5. **겹치는 목표** — 단기(다음 스테이지)·중기(영웅 승급)·장기(도감 완성) 목표가 항상 동시에 걸려 있게 설계한다. "지금 그만둘 타이밍"이 생기지 않아야 한다.

### 삼국지 IP 활용
- 영웅은 실제 삼국지 인물 기반(관우, 조조, 여포, 제갈량 등). 등급·스탯은 연의(演義)의 위상을 반영해 수집 만족감을 높인다.
- 스테이지는 황건적의 난 → 적벽대전 등 실제 사건 순서를 따라 진행감을 부여한다.

## 3. 기술 규약

- **바닐라 JS ES모듈**. 빌드 도구·번들러·프레임워크 금지. `<script type="module">`로 로드.
- **PWA**: `manifest.json` + `sw.js`(서비스워커). 전체 에셋 오프라인 캐시 — 비행기 모드에서도 완전히 플레이 가능해야 한다.
- **외부 의존성 금지**: CDN·외부 폰트·외부 이미지 사용 불가(오프라인 보장). 아트는 CSS/SVG로만.
- **저장**: `localStorage`에 JSON 세이브. 세이브 스키마 최상위에 `version` 필드 필수. 스키마가 바뀌면 version을 올리고 마이그레이션 함수를 추가한다(기존 유저 세이브가 깨지면 안 됨). 이 구조가 추후 서버 동기화의 기반이 된다.
- **모바일 우선**: 세로 화면(포트레이트) 기준 레이아웃, 터치 타깃 최소 44px, 하단 탭 내비게이션. 데스크톱은 모바일 뷰 중앙 배치로 대응.
- **게임 루프**: 화면 갱신은 `requestAnimationFrame`, 방치·전투 진행 계산은 실제 경과 시간(타임스탬프) 기반. 탭이 백그라운드여도 계산이 어긋나지 않아야 한다.

## 4. 파일 구조

```
index.html          # 단일 진입점 (화면 컨테이너만, 로직 없음)
manifest.json       # PWA 매니페스트 (홈 화면 설치)
sw.js               # 서비스워커 (오프라인 캐시)
css/                # 스타일 (base, 화면별, 연출)
js/
  core/             # 엔진: state.js(상태), save.js(저장/마이그레이션),
                    #       loop.js(게임루프), events.js(이벤트 버스)
  systems/          # 게임 로직: battle.js, gacha.js, growth.js, offline.js
  ui/               # 화면·연출: 화면별 렌더러, effects.js(파티클·카운트업)
  data/             # 콘텐츠·밸런스: heroes.js, stages.js,
                    #               gacha-tables.js, balance.js
```

### 데이터 주도 원칙 (가장 중요)
- **게임 콘텐츠와 모든 밸런스 숫자는 `js/data/`에만 존재한다.** 영웅 추가, 확률 조정, 성장 곡선 변경은 데이터 파일 수정만으로 끝나야 한다. systems/ 코드에 매직 넘버가 박혀 있으면 잘못된 것이다.

## 5. 코딩 규칙

- **상태 단일 소스**: 모든 게임 상태는 `core/state.js`가 소유한다. 상태 변경은 반드시 state가 제공하는 함수를 통해서만 하고, UI는 이벤트 버스로 변경을 구독해 갱신한다. UI가 상태를 직접 고치지 않는다.
- **모듈 경계**: `data/`는 순수 데이터(로직 없음), `systems/`는 로직(DOM 접근 없음), `ui/`만 DOM을 만진다. 경계를 넘는 import는 금지.
- **밸런스 변경 이력**: `balance.js` 수정 시 파일 상단 주석에 날짜·변경 사유를 한 줄 남긴다.
- **연출 우선순위**: 기능이 돌아가는 것과 "손맛 나게" 돌아가는 것은 다르다. 중독성 5대 장치에 해당하는 기능은 연출까지가 완성이다.

## 6. 작업 워크플로

1. 수석이 단계별 프롬프트로 작업을 지시한다.
2. 선임은 구현 전 영향 범위(세이브 스키마 변경 여부 포함)를 확인하고, 스키마가 바뀌면 마이그레이션을 함께 구현한다.
3. 매 단계 완료 시 **브라우저에서 실제 동작을 확인한 후** 완료를 보고한다. 보고에는 무엇이 동작하는지와 수석이 직접 확인하는 방법(열 파일, 누를 버튼)을 포함한다.
4. 게임 디자인 헌장(2장)에 어긋나는 지시가 있으면 구현 전에 근거를 들어 이견을 보고한다 — 최종 결정은 수석이 한다.
